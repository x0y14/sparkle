# WebGPU チャートライブラリ クリーンルーム設計指示書

この文書は、WebGPU を用いた高性能チャートライブラリを新規実装するための設計指示書である。
実装者はこの文書に従って設計・実装を行う。各セクションは「目的→方針→理由→代替案→手順」の順で記述されている。
定数値・データレイアウト・命名などの実装詳細は実装者が自ら決定する。本文書はそのための判断基準を提供する。

### 各層の責務と実装者が守るべき層間の契約

**データ管理層**
- 入力: 任意形式のデータ + 変換パラメータ（オフセット等）
- 出力: GPU バッファ（読み取り可能な状態）
- 責務: 形式変換、GPU バッファ確保・成長、変更検出、ストリーミング追加
- 他層との関係: コーディネーターから「このデータを GPU に載せて」と指示される。GPU 描画層はこの層が管理するバッファを参照して描画する

**GPU 描画層**
- 入力: GPU バッファ（データ管理層から）、変換行列（コーディネーターから）、スタイル設定（設定正規化層から）
- 出力: Canvas 上のピクセル
- 責務: チャートタイプ別の描画ロジック、シェーダー管理、MSAA テクスチャ管理、レンダーパス構成
- 内部構造: チャートタイプごとに独立した描画モジュール + 共有基盤（グリッド、軸線、クロスヘア、ハイライト）
- 他層との関係: コーディネーターが「この変換行列でこのバッファを描け」と指示する

**インタラクション層**
- 入力: ブラウザのポインターイベント
- 出力: ズーム状態の変更、ヒットテスト結果、クロスヘア位置
- 責務: イベント正規化、ズーム/パン計算、最近傍点検索
- 他層との関係: コーディネーターに「ズームが変わった」「ポインターがこの点の近くにいる」と通知。コーディネーターはこれを受けて再描画・ツールチップ更新をトリガーする

**DOM オーバーレイ層**
- 入力: ラベルテキスト・位置（コーディネーターから）、ツールチップ内容・位置
- 出力: HTML 要素の生成・更新
- 責務: 軸ラベル、タイトル、ツールチップ、凡例の DOM 操作
- 他層との関係: コーディネーターが「このテキストをこの位置に表示して」と指示する

### 1フレームのライフサイクル（情報フロー全体像）

```
ユーザー操作（ズーム/データ追加/設定変更）
  ↓ dirty フラグを立てる
requestAnimationFrame
  ↓
コーディネーターが dirty を検知
  ↓
[レイアウト] マージン + Canvas サイズ → プロット領域の矩形
  ↓
[スケール] データバウンド + プロット領域 → 変換関数
  ↓
[ズーム] ズーム状態(%) + データバウンド → 表示ドメイン → スケール更新
  ↓
[データ] 表示ドメインでデータをスライス → データ管理層に転送指示
  ↓
[GPU描画] スケール→変換行列、データバッファ→描画コマンド生成→GPU実行
  ↓
[DOM] スケール + ティック値 → 軸ラベル配置
       ヒットテスト結果 → ツールチップ更新
  ↓
フレーム完了
```

### この構造の理由と代替案

**理由:**
- コーディネーターが中央にいること: レイアウト→スケール→ズーム→データ→描画の実行順序を1箇所で保証するため
- 各層を独立させること: 新チャートタイプ追加時にデータ管理層やインタラクション層の変更が不要になるため
- DOM層を分離すること: DOM操作はレイアウトスラッシングを起こしうるため、GPU描画パイプラインとは独立したタイミングで制御する必要があるため

**代替案と棄却理由:**
- 各レンダラーが自分でスケール計算・データ取得を行う分散型 → フレーム内の実行順序が保証できず、レイアウトとスケールの不整合が起きる
- イベント駆動（Pub/Sub）で層間を接続 → フレーム内の処理順序が暗黙的になり、デバッグが困難。描画パイプラインでは明示的な順序制御が重要

## 第1章: このライブラリが解決する問題

### 描画技術の選定

**目的:** ブラウザ上で数十万〜数百万ポイントのチャートを 60fps で描画する。

**方針:** WebGPU を描画基盤として採用する。

**理由:**
- コンピュートシェーダーが使える（密度マップの2Dビニングに必要）
- バッファの部分書き込みなどリソース管理を細かく制御できる（ストリーミングデータに有利）
- ブラウザベンダーが注力しており、今後のパフォーマンス最適化が期待できる

**代替案と棄却理由:**
- Canvas 2D: CPUバウンド。CPUが1ポイントずつラスタライズするため、100万ポイントでフレーム時間が数百msに達する
- SVG: 各データポイントがDOM要素になるため、数万ポイントでレイアウトエンジンが破綻する
- WebGL: コンピュートシェーダーがなく、密度マップ等の処理をフラグメントシェーダーで模倣する必要がある。リソース管理もWebGPUほど明示的でない

### サポートするチャートタイプと各タイプの描画上の特性

| チャートタイプ | 描画上の特性 | 主な技術的機能 |
|-------------|-----------|-------------|
| 折れ線 | 連続した線分の列 | 可変太さの線を GPU で描く方法、アンチエイリアシング |
| 棒 | 矩形の集合 | 複数シリーズのレイアウト（クラスタリング・スタッキング） |
| エリア | 折れ線 + ベースラインで囲まれた領域の塗りつぶし | 帯状領域の三角形分割 |
| 散布図 | 独立した円の集合 | 大量描画時の性能、密度可視化への切替 |
| 円グラフ | 扇形の集合 | 角度ベースの SDF で内外判定 |
| ローソク足 | ボディ（矩形）+ ヒゲ（線）の複合図形 | 1データ点から複数の幾何形状を生成 |

### 必要な機能

- ズーム/パン（ホイール、ドラッグ、ピンチ、スライダーUI）
- ツールチップ（最近傍データ点の情報表示）
- 凡例（シリーズの表示/非表示切替）
- リアルタイムデータ追加（ストリーミング）
- 複数チャートの同期（ズーム位置・クロスヘアの連動）

## 第2章: WebGPU で 2D チャートを描く際の技術的機能

### 機能A: 線の描画

**目的:** 任意の太さの線分を GPU で高速に描画する。

**方針:** 各線分セグメントを2三角形で構成された四角形（quad）として描く。四角形の幅は「線の太さ + AA用マージン」とする。セグメント数分の GPU インスタンスを発行し、頂点シェーダー内で四角形に展開する。

**理由:**
- GPU は三角形の描画に最適化されている
- インスタンシングにより1回の draw call で全セグメントを描画できる
- 四角形の幅をフラグメントシェーダーと組み合わせることでサブピクセル精度の AA を実現できる

**代替案と棄却理由:**
- line-list プリミティブ: WebGPU の line-list は太さが常に1デバイスピクセルで制御不可。チャートの線幅要件を満たせない
- CPU 側でラスタライズして画像テクスチャとして送る: GPU の並列性を活かせず、ズーム/パンのたびに再ラスタライズが必要
- データ空間で四角形を展開: X軸とY軸のスケールが異なるためアスペクト比の歪みで太さが不均一になる。画面空間で展開すること

**実装手順:**
1. セグメント数 (N-1) 分のインスタンスを発行する（N = ポイント数）
2. 頂点シェーダーで instance_index から隣接2点を読み出す
3. 2点をクリップ空間に変換し、さらに画面空間（デバイスピクセル）に変換する
4. セグメントの方向ベクトルと法線ベクトルを計算する
5. 法線方向に「線の太さ/2 + AAマージン」だけ膨らませて四角形（6頂点）を生成する
6. 画面空間からクリップ空間に戻す
7. フラグメントシェーダーで法線方向の距離に基づく SDF AA を計算する

### 機能B: アンチエイリアシング

**目的:** 描画されるすべての図形のエッジを滑らかにする。

**方針:** 2層のアンチエイリアシングを組み合わせる:
1. MSAA: レンダーパイプラインの sampleCount 設定で有効化する。ポリゴンの幾何学的エッジに作用する
2. SDF AA: フラグメントシェーダー内で「境界からの符号付き距離」を計算し、smoothstep で alpha をフェードする。図形の見た目のエッジに作用する

**理由:** 両者は相補的に働く。MSAA は四角形のジオメトリエッジを滑らかにし、SDF AA は四角形内部の「線の太さの境界」や「円の輪郭」を滑らかにする。

**代替案と棄却理由:**
- MSAA のみ: 四角形のエッジは滑らかになるが、四角形内部で線の太さの境界がカクカクになる
- SDF AA のみ: 四角形の幾何学的エッジ（線の端点付近）にジャギーが残る
- FXAA などのポストプロセス AA: 追加パスが必要で、SDF AA ほど精密な制御ができない。テキストがぼやける問題もある

### 機能C: データ座標からピクセルへの変換パイプライン

**目的:** ユーザーデータ（時刻と株価など）を GPU のクリップ空間 [-1,1]² を経由して画面ピクセルに正しく変換する。float32 の精度制約を克服する。

**方針:** CPU 側で基準点（オフセット）を決め、各データ値からオフセットを引いた相対値を float32 に変換して GPU に送る。変換行列にオフセットを折り込むことで、GPU 側では小さな値の範囲で計算を完結させる。

**理由:**
- GPU は float32 しか扱えず、大きな値（エポックミリ秒は約 1.7×10¹² のオーダー）を直接扱うと精度が失われる
- オフセットを引いた相対値なら float32 の仮数部（約7桁）で十分な精度を確保できる
- 変換行列を1回送ればシェーダー内で全頂点に一括適用できる

**代替案と棄却理由:**
- 行列ではなく個別のユニフォーム（scaleX, offsetX 等）を送る → 行列なら1回の mat×vec で完結し、GPU の行列演算ユニットを活用できる
- CPU 側で全頂点を画面座標に変換してから送る → ズーム/パンのたびに全データの再計算・再アップロードが必要

**実装手順:**
```
ユーザーデータ（float64）
  → CPU: オフセットを減算 → 相対値（float32 で精度十分）
  → GPU バッファに書込み
  → 頂点シェーダー: 変換行列（オフセット込み）で クリップ空間へ
  → 画面ピクセル
```

### 機能D: CPU と GPU の役割分担

**方針:** 以下の通り処理を CPU と GPU に分担する。

| 処理 | 担当 | 理由 |
|------|------|------|
| 幾何学的描画（線、矩形、円） | GPU | 大量の並列処理に最適 |
| テキスト描画 | DOM/CSS | GPU でのフォントラスタライズは極めて困難。ブラウザの DOM/CSS はカーニング、合字、多言語を完璧にサポート |
| ヒットテスト（最近傍データ点の検索） | CPU | GPU からの読み戻しは遅い。CPU のキャッシュに載る程度のデータなら CPU の方が速い |
| ツールチップ・凡例 | DOM/HTML/CSS | 数が少なくDOM操作で十分高速。CSSのレイアウト機能を自然に活用できる |

## 第3章: 座標系と変換の設計

### 4つの座標空間

以下の4つの座標空間を定義し、実装全体で一貫して使用する。

1. **データ空間**: ユーザーの値（例: x = エポックミリ秒, y = 株価）
2. **クリップ空間**: GPU の正規化デバイス座標 [-1,1]²。GPU はこの空間で頂点を受け取る
3. **デバイスピクセル空間**: Canvas の物理ピクセル。Canvas の width/height 属性の値
4. **CSS ピクセル空間**: ブラウザのレイアウトピクセル。clientWidth/clientHeight の値

### 各変換の数学

**データ空間 → クリップ空間**

アフィン変換（線形スケール）で domain（データ範囲）を range（クリップ範囲）に写す:
```
t = (value - domainMin) / (domainMax - domainMin)
result = rangeMin + t * (rangeMax - rangeMin)
```

**方針:** この変換を 4×4 行列にエンコードし、GPU ユニフォームとして送る。

**理由:**
- GPU ユニフォームとして1回送ればシェーダー内で全頂点に一括適用できる
- ズーム/パン時は行列だけ更新すればよく、データの再アップロードが不要
- 4×4 行列は GPU のハードウェアが最も効率的に処理できる形式

**代替案と棄却理由:**
- 個別ユニフォーム（scaleX, offsetX 等）: 行列なら1回の mat×vec で完結し、GPU の行列演算ユニットを活用できる。個別ユニフォームでは演算回数が増える
- CPU 側で全頂点を変換: ズーム/パンのたびに全データの再計算・再アップロードが必要になる

**クリップ空間 → 画面ピクセル**

```
screenX = (clipX + 1) * 0.5 * canvasWidth
screenY = (1 - clipY) * 0.5 * canvasHeight   ← Y軸は反転（クリップ空間は上が+1、画面は上が0）
```

**CSS ピクセル → デバイスピクセル**

```
devicePx = cssPx * devicePixelRatio
```

### プロット領域の定義

**目的:** Canvas 全体のうち、チャートデータを描画する矩形領域を確定する。

**方針:** Canvas 全体からマージン（上下左右）を引いた矩形を「プロット領域」と定義する。マージンは CSS px で指定する（軸ラベルの幅に依存するため）。プロット領域の四隅をスケールの range（出力範囲）とする。

### ズーム状態の管理

**目的:** データの型（数値/時刻/カテゴリ）に依存しない統一的なズーム操作を実現する。

**方針:** ズーム状態をパーセント空間 [0, 100] で管理する。0% = データの先頭、100% = データの末尾。

**理由:**
- データ型に依存しないため、すべてのチャートタイプで同じズームロジックを使える
- ズームスライダーUIとの対応が自然になる

**代替案と棄却理由:**
- データドメイン値で直接管理: 時刻型・カテゴリ型でズーム幅の計算方法が異なり、ロジックが型ごとに分岐する
- ピクセル空間で管理: Canvas サイズ変更時にズーム状態が変わってしまう

**実装手順:**
1. ズームウィンドウを [startPct, endPct] で保持する（初期値は [0, 100]）
2. パーセント → データドメインの変換: domainValue = domainMin + (percent / 100) * (domainMax - domainMin)
3. ドメイン → パーセントの逆変換: percent = (value - domainMin) / (domainMax - domainMin) * 100
4. ズーム変更のたびにスケールの domain を更新し、変換行列を再計算する

### 逆変換

ヒットテスト（画面上のクリック位置からデータ値を求める）には逆変換を使用する:
```
CSS px → デバイスピクセル → クリップ空間 → データ空間
```
各変換はアフィン変換なので、逆関数を簡単に求められる。

## 第4章: CPU → GPU データ転送の設計

### 入力データの多様性への対応

**目的:** ユーザーが渡す様々な形式のデータを、GPU が受け取れる連続バイト列に変換する。

**方針:** 入力形式は複数受け入れるが、GPU 転送前に統一形式にパックする。形式に依存しない抽象的なアクセサ層（「i番目のxを取得」「i番目のyを取得」のような操作）を設け、パック処理はこのアクセサ経由で行う。

**理由:** アクセサ層を挟むことで、新しい入力形式の追加がパック処理に影響しない。

**代替案と棄却理由:**
- 入力形式ごとに個別のパック処理を書く → 形式が増えるたびにパック処理のコードが増殖し、保守が困難になる
- ユーザーに統一形式での入力を強制する → API の使い勝手が悪くなり、ユーザー側で変換コードが必要になる

**実装手順:**
1. アクセサの種類を定義する: ポイント数取得、x値取得、y値取得、オプショナルなサイズ値取得、バウンド計算、nullギャップ検出
2. サポートする入力形式ごとにアクセサを実装する:
   - オブジェクト配列 `[{x, y}, ...]` やタプル配列 `[[x, y], ...]`
   - 列方向配列 `{x: Float32Array, y: Float32Array}`
   - インターリーブ配列 `Float32Array([x0, y0, x1, y1, ...])`
   - OHLC（ローソク足用）: 1ポイントあたり5値（時刻, 始値, 高値, 安値, 終値）
3. アクセサ経由で統一形式にパックし、GPU バッファに書き込む

**注意事項:**
- アクセサは GPU アップロードのたびに呼ばれるため、分岐を最小限にしてパフォーマンスを確保すること

### GPU バッファの管理戦略

**目的:** ストリーミングデータ追加時の GPU バッファ再確保コストを最小化する。

**方針:** バッファ容量を2のべき乗で幾何成長させる。追加時に容量内なら追加分だけ書き込み、容量を超えたら倍のサイズで再確保する。

**理由:** N 回の追加に対して再確保は O(log N) 回。償却計算量は追加1回あたり O(1)。

**代替案と棄却理由:**
- 必要サイズぴったりで毎回再確保: 追加のたびに alloc/dealloc が発生し、O(N²) の累積コスト
- 固定サイズの大きなバッファを最初に確保: 最大データ量が予測できない場合にメモリを浪費するか不足する
- リングバッファ: 古いデータを上書きする場合には適するが、全データを保持したいチャートには不適

**実装手順:**
1. 必要バイト数を計算する
2. 2のべき乗に切り上げる
3. WebGPU の制約に従い4バイトの倍数にアライメントする
4. 現バッファの容量以下ならそのまま末尾に書き込む
5. 超過したら新バッファを確保し、全データを転送する

**注意事項:**
- WebGPU はバッファサイズに4バイトアライメントを要求する
- バッファ作成時に usage フラグ（VERTEX, STORAGE, COPY_DST 等）を正しく指定すること

### 変更検出

**目的:** 同一データでの冗長な GPU 転送をスキップする。

**方針:** データのバイト列に対する高速な非暗号学的ハッシュを計算し、前回値と比較する。一致すれば転送をスキップする。ストリーミング追加時はインクリメンタルにハッシュを更新する。

**理由:** ハッシュ比較は O(1)。全バイト比較は O(N) で、データが同一の場合でも毎回全走査が必要。

**代替案と棄却理由:**
- バイト列の全比較: 同一データでも O(N) かかる。ストリーミングで毎フレーム呼ばれると無視できないコスト
- dirty フラグのみ: ユーザーが設定更新で同じデータを渡した場合に検出できない
- 暗号学的ハッシュ（SHA-256 等）: 衝突耐性は不要でオーバースペック。速度重視で非暗号学的ハッシュを選ぶ

**注意事項:**
- データを整数列（uint32）として扱いハッシュする。浮動小数点のビットパターンをそのまま使えば NaN 等も区別できる
- インクリメンタル更新可能なハッシュアルゴリズムを選ぶこと（追加分だけ処理すれば全体のハッシュが得られるもの）

### Float32 精度問題

**目的:** float64 のユーザーデータを float32 の GPU バッファに精度を損なわず転送する。

**方針:** シリーズごとに「基準点（オフセット）」を CPU 側で決め（例: 最初のデータ点の x 値）、オフセットを引いた相対値を float32 に変換する。レンダリング時の変換行列にオフセットを復元する。

**理由:** エポックミリ秒（~1.7×10¹²）のような大きな値では float32 の仮数部（約7桁）が不足し、数百〜数千の精度落ちが生じる。オフセットを引いた相対値なら精度が確保できる。

**代替案と棄却理由:**
- float64 対応の GPU バッファを使う → WebGPU は float64 をネイティブサポートしていない
- データを正規化して [0, 1] 範囲に収める → 逆変換が複雑になり、ヒットテスト時の精度に問題が出る

**実装手順:**
```
元データ x（float64、例: 1710000000000）
  → CPU: xRelative = x - offset（float64 で計算）
  → float32 に変換（相対値なので精度十分）
  → GPU バッファに書込み
  → 頂点シェーダー: 変換行列（CPU が float64 で offset を折り込み済み）を適用
  → クリップ空間座標（正しい位置）
```

**注意事項:**
- オフセットの選び方: 最初のデータ点、データの中央値、最小値など。データ全体が float32 の精度に収まる範囲になればよい
- 変換行列の計算は CPU 側で float64 で行い、結果の行列要素を float32 として GPU に送る

## 第5章: チャートタイプ別の描画戦略

### 折れ線グラフ

**目的:** 可変太さの線を GPU で高速に描画する。

**方針:** インスタンシング + 画面空間での四角形展開を用いる。詳細な手順は第2章機能Aを参照すること。

**注意事項:**
- null ギャップの扱い: NaN を使って「この点は描画しない」を表現する。シェーダーが NaN を検出したらデジェネレート四角形（面積ゼロ）を出力すれば、その区間は描画されない
- NaN の検出は IEEE 754 の性質（NaN ≠ NaN）を利用できる

### 棒グラフ

**目的:** 複数シリーズのバーをクラスタリングまたはスタッキングで配置し描画する。

**方針:** バーの矩形を CPU 側でクリップ空間の座標に事前計算し、GPU にはクリップ空間の四角形としてそのまま渡す。インスタンシングで1バー = 1インスタンス = 6頂点の四角形として描画する。

**理由:**
- クラスタリング: 同じ x 位置に複数シリーズのバーを並べるレイアウト。シリーズ間の位置関係は GPU で並列処理しにくい（シリーズ数とスタック構成の情報が必要）
- スタッキング: 同じスタック ID のシリーズの y 値を累積する。前のシリーズの結果に依存する逐次処理

**代替案と棄却理由:**
- GPU 側でレイアウト計算 → シリーズ間依存があるため並列化困難。正しいレイアウトを保証するにはシリーズ間の同期が必要で棄却

**実装手順:**
1. データとシリーズ設定（スタックID、バー幅等）を受け取る
2. CPU でクラスタリング・スタッキングのレイアウト計算を行う
   - クラスタリング: 1カテゴリの幅を等分し、シリーズごとにスロットを割り当てる
   - スタッキング: 正値と負値を別々に累積する（正値は上方向、負値は下方向）
3. 各バーのクリップ空間座標（x, y, width, height）+ 色を計算する
4. GPU バッファに書き込む
5. インスタンシングで描画する

### エリアグラフ

**目的:** 折れ線とベースラインで囲まれた領域を塗りつぶす。

**方針:** トライアングルストリップで各データ点を「上端（データ値）」と「下端（ベースライン）」に二重化する。

**理由:** 各データ点から2頂点を交互に並べるだけで、帯状の領域が自動的に三角形列に分割される。

**代替案と棄却理由:**
- triangle-list + インデックスバッファ → strip なら追加のインデックスバッファ不要で簡潔。メモリ使用量も少ない

**実装手順:**
1. 頂点シェーダーで vertex_index の偶奇により y をデータ値かベースラインに切り替える
2. トライアングルストリップが自動的に帯状に解釈される
3. フラグメントシェーダーでフラットカラーで塗りつぶす

**注意事項:**
- null ギャップ: NaN ポイントに対してデジェネレート頂点ペア（面積ゼロの三角形）を挿入して塗りを途切れさせる
- 頂点数 = 2 × ポイント数

### 散布図（ポイントモード）

**目的:** 大量の独立した円を高速に描画する。

**方針:** 各ポイントに対して四角形をインスタンシングで生成し、フラグメントシェーダーで円の SDF を評価する。

**理由:** SDF なら四角形（6頂点）の中でピクセル単位の距離計算により完璧な円を描ける。アンチエイリアシングも境界付近の smoothstep で自然に得られる。

**代替案と棄却理由:**
- 三角形で円を近似 → 頂点数膨大。滑らかな円を得るにはポイントあたり数十頂点が必要で棄却

**実装手順:**
1. 各ポイントの座標 + 半径をバッファに書き込む
2. インスタンシングで四角形を展開する
3. フラグメントシェーダーで中心からの距離を計算する
4. distance > radius なら discard（円の外）
5. 境界付近で smoothstep でフェードする（AA）

### 散布図（密度モード）

**目的:** 数百万ポイントを密度ヒートマップとして可視化する。

**方針:** コンピュートシェーダーで2Dビニング → フラグメントシェーダーでカラーマップ適用、の2段パイプラインで実装する。

**理由:** 2Dビニング（各ビンに落ちるポイント数をカウント）はアトミック操作が必要であり、WebGPU のコンピュートシェーダーがこれをサポートしている。

**代替案と棄却理由:**
- フラグメントシェーダーのみでビニング → アトミック操作が使えず、正確なカウントが不可能で棄却
- CPU 側ビニング → 数百万ポイントの2D集計は CPU では遅く棄却

**実装手順:**
1. コンピュートシェーダー①: プロット領域をグリッドに分割し、各ビンに落ちるポイント数をアトミックカウントする
2. コンピュートシェーダー②: ビンの最大値を計算する
3. フラグメントシェーダー: 各ビンのカウントを正規化し、カラーマップ LUT で色に変換する

**注意事項:**
- 正規化モード: 対数、平方根、線形の3モードを用意する。データの分布に応じて選択する（対数は極端な外れ値がある場合に有用）
- カラーマップ: 正規化値 [0,1] を色に変換する1D ルックアップテーブルテクスチャ（例: 256×1ピクセル）を用いる

### 円グラフ

**目的:** 扇形を GPU で描画する。ドーナツチャートにも対応する。

**方針:** 各スライスを四角形としてインスタンシングし、フラグメントシェーダーで「半径方向」と「角度方向」の2つの SDF で内外判定する。

**理由:** SDF なら頂点数が少なく（スライスあたり6頂点）、かつ完璧な曲線を描ける。

**代替案と棄却理由:**
- 三角形メッシュで扇形近似 → スライス数×分割数の頂点が必要で棄却。曲線の滑らかさも分割数に依存してしまう

**実装手順:**
1. スライスごとの角度範囲 + 内径/外径をバッファに書き込む
2. インスタンシングで四角形を展開する
3. フラグメントシェーダーで:
   a. フラグメント位置の角度を atan2 で求め、スライスの開始角〜終了角の範囲内かを判定する
   b. 中心からの距離が内径〜外径の範囲内かを判定する
   c. 両方の条件を smoothstep で合成して AA 付きの alpha を計算する

**注意事項:**
- ドーナツチャート対応: 内径を0より大きく設定すればドーナツになる
- 角度の wrap-around に注意: atan2 の値域と開始角/終了角の関係を正しく処理すること

### ローソク足

**目的:** 1つのデータ点（時刻, 始値, 高値, 安値, 終値）からボディとヒゲを描画する。

**方針:** 1インスタンスあたり3四角形分の頂点（6×3 = 18頂点）を発行し、vertex_index の範囲でボディとヒゲを切り替える。

**理由:** 18頂点なら1つのシェーダーで完結し、draw call を最小化できる。

**代替案と棄却理由:**
- ボディとヒゲを別インスタンス → draw call かバッファ管理が複雑化で棄却

**実装手順:**
1. OHLC データ（5値/ポイント）を受け取る
2. CPU でクリップ空間座標に変換し、色（上昇/下降）を決定する
3. GPU バッファに書き込む
4. 頂点シェーダーで vertex_index の範囲から3つの四角形に振り分ける:
   - ボディ: open〜close の範囲
   - 上ヒゲ: ボディ上端〜high
   - 下ヒゲ: low〜ボディ下端

## 第6章: アンチエイリアシングの統合設計

### 2層のアンチエイリアシング

**目的:** MSAA と SDF AA を組み合わせ、すべての図形のエッジを滑らかに描画する。

**方針:**
- 第1層 MSAA: レンダーパイプラインの sampleCount を設定して有効化する。サンプル数は4を用いる（性能と品質のバランスが良い）
- 第2層 SDF AA: フラグメントシェーダーで「境界からの距離」を計算し、smoothstep で alpha をフェードする

### SDF AA の共通パターン

**方針:** すべての図形（線、円、扇形、リング）で以下の共通手順を適用する:
1. 境界からの符号付き距離 (signed distance) を計算する
2. fwidth() で画面空間の距離勾配を取得する
3. smoothstep(0, fwidth(dist), dist) で1ピクセル幅のフェードをかける

**理由:** fwidth() を使うことで、ズームレベルやディスプレイ解像度が変わっても常に「画面上の1ピクセル幅」でフェードできる。

**代替案と棄却理由:**
- 固定幅のフェード: 高DPIでフェードが細すぎ、ズームアウト時に太すぎる
- discard ベースのハードエッジ: ジャギーが目立つ

### 3パスレンダリング

**目的:** 更新頻度の異なる描画要素を独立に再描画可能にし、不要な再描画を最小化する。

**方針:** 描画を3パスに分離する:
- パス1: メインシーン（MSAA有効）→ リゾルブ
- パス2: パス1結果をBlit + アノテーション（MSAA有効）→ リゾルブ
- パス3: パス2結果 + インタラクションオーバーレイ → スワップチェーン

**理由:** マウス移動時（毎フレーム）にはパス3のみ、アノテーション変更時にはパス2+3のみ、データ変更時にはパス1+2+3を実行すればよい。

**代替案と棄却理由:**
- 1パス: すべてを同じテクスチャに描くため、クロスヘアの移動だけで全チャートデータの再描画が必要
- 2パス（メインシーン + オーバーレイ）: アノテーションにもMSAAが必要だが、メインのMSAAテクスチャに追記するとアノテーション更新にメイン再描画が必要

**実装手順:**

**パス1: メインシーン**
- 対象: 背景、グリッド線、全チャートデータ（折れ線、棒、エリア、散布図等）、軸線
- MSAA 有効 → リゾルブして1xテクスチャに格納する
- データが変わらなければ再描画不要（キャッシュ可能）

**パス2: アノテーションオーバーレイ**
- パス1のリゾルブ結果をテクスチャとして読み取り、画面にBlit（ピクセル正確コピー）する
- その上にアノテーション（マーカー、リファレンスライン等）を重ね描きする
- MSAA 有効（アノテーションにもAA が必要）
- アノテーション更新時はこのパスだけ再実行する（パス1は不要）

**パス3: インタラクションオーバーレイ**
- パス2のリゾルブ結果の上にクロスヘア、ハイライトリングを描画する
- MSAA 不要（これらの要素は SDF AA で十分）
- 最終出力をスワップチェーンに書き込む
- マウス移動のたびに更新されるが、このパスだけで済む

## 第7章: ユーザー操作の情報フロー

### ホイールズーム

**目的:** ホイール操作でユーザーが注目している箇所を中心にスムーズにズームする。

**方針:**
1. ブラウザのホイールデルタを正規化する（ピクセル/行/ページの3単位を統一）
2. 極端な値をクランプする
3. ズームファクターを指数関数 exp(delta * sensitivity) で計算する
4. カーソル位置をアンカーとしてパーセント空間のズームウィンドウを更新する
5. パーセント → データドメインに変換する
6. 新しいドメインから変換行列を再計算する
7. 次フレームで再描画する

**理由:**
- 指数関数: 一定のスクロール量で一定の「倍率」が得られる。線形だとズームイン/アウトで体感速度が異なる
- カーソルアンカー: ユーザーが注目している点がズーム後も画面上の同じ位置に留まり、自然な操作感になる

**代替案と棄却理由:**
- 線形ファクター（factor = 1 + delta * k）: ズームイン（×2）とズームアウト（×0.5）で同じデルタ量にならず、操作感が非対称
- 固定倍率のステップズーム: スムーズさに欠ける
- 画面中央をアンカー: ユーザーの視線とズーム中心がずれ、操作のたびに目的の箇所を探す必要が生じる

### パン（ドラッグ）

**目的:** ドラッグ操作で表示範囲を水平にスライドする。

**方針:**
1. ドラッグ中のポインター移動量（CSS px）を取得する
2. 移動量をパーセント空間に変換する: 移動量(%) = -(移動ピクセル / プロット幅) × 現在の表示幅(%)
3. ズームウィンドウの start/end を同量だけシフトする（span は不変）
4. [0, 100] にクランプする

### タッチ操作

**目的:** タッチデバイスでのズームとパンを実現する。

**方針:**
- シングルフィンガー: パンと同じ処理を行う
- ツーフィンガーピンチ: 2指間距離の変化をズームファクターに変換する。2指の中点をアンカーとする

### タップ vs ドラッグの区別

**目的:** タッチデバイスでタップ（選択）とドラッグ（パン）を正しく区別する。

**方針:** pointerdown で候補を記録し、pointerup で「距離が閾値以下」かつ「経過時間が閾値以下」ならタップと判定する。それ以外はドラッグとして処理する。

### ヒットテスト

**目的:** マウス位置から最も近いデータ点を見つけてツールチップを表示する。

**方針:**
1. ポインター位置（CSS px）がプロット領域内か判定する（マージンを差し引く）
2. CSS px → データ空間に逆変換する（スケールの逆関数）
3. データ空間で最近傍点を検索する
   - X が単調増加なら二分探索（O(log n)）を用いる
   - そうでなければ線形探索（O(n)）を用いる
4. 最近傍点が閾値距離内なら採用する
5. 最近傍点の情報でツールチップを更新する

**理由:** 二分探索は時系列データ（x が単調増加）で O(log n) の高速検索を実現する。距離計算は二乗距離で比較し、最終結果にのみ平方根を取る（平方根は遅いため）。

**代替案と棄却理由:**
- GPU でヒットテストを行う → GPU からの読み戻しは遅い。CPU のキャッシュに載る程度のデータなら CPU の方が速い
- 全点に対して常に線形探索 → 時系列データでは二分探索の方が圧倒的に速い

**注意事項:**
- 棒グラフは距離ベースではなく、矩形内にカーソルがあるかで判定する
- 円グラフは角度と半径で判定する
- 散布図は半径を考慮して許容距離を広げる

### 複数チャートの同期

**目的:** 複数のチャートで X 軸のズーム位置やクロスヘアの位置を連動させる。

**方針:** あるチャートのインタラクション結果（ズーム範囲やクロスヘアの X データ値）を他のチャートに伝播する API を提供する。伝播先は受け取ったデータ値から自身のスケールで画面座標を再計算する。

## 第8章: DOM と GPU の責務分担

### テキスト描画

**目的:** チャート内のテキスト要素（軸ラベル、タイトル、ツールチップ、凡例）を描画する。

**方針:** すべてのテキスト描画を DOM/CSS に委譲する。Canvas 上に DOM オーバーレイ層を重ねて配置する。

**理由:** チャートのテキスト要素は数が少なく（数十個程度）、DOM で十分高速。

**代替案と棄却理由:**
- GPU でテキスト描画（フォントアトラス方式）: フォントのラスタライズ、カーニング、合字、多言語対応を自前で実装する必要があり、極めて複雑
- Canvas 2D のテキスト描画を別レイヤーで使用: 2つの Canvas を管理する複雑さが増す。DOM の方がレイアウト制御（折り返し、overflow等）が自然

### Canvas と DOM オーバーレイの重畳

**目的:** Canvas のチャート描画と DOM のテキスト要素を正確に重ね合わせる。

**方針:** コンテナ要素を `position: relative` にし、Canvas と DOM オーバーレイ層を `position: absolute` で重ねる。

**理由:**
- Canvas と DOM 要素の座標系が一致する（同じコンテナ内なので）
- GPU レンダリングと DOM 操作を独立に制御できる

**代替案と棄却理由:**
- Canvas を DOM フローに含めて兄弟要素として配置 → 座標系の一致が保証されず、リサイズ時にずれが生じる

**注意事項:**
- オーバーレイ層は `pointer-events: none` でポインターイベントを透過させる
- インタラクティブな要素（凡例のクリック等）のみ `pointer-events: auto` に設定する

### 軸ラベルの配置

**目的:** ズームレベルに応じてラベルの数と書式を動的に調整する。

**方針:** 利用可能な幅（プロット領域の幅）に何個のラベルが重ならずに入るかを計算し、最大数を選択する。

**実装手順:**
1. ラベルの幅を実際の文字列をレンダリングして測定する
2. プロット領域の幅をラベル幅で割り、最大ラベル数を決定する
3. 時間軸の場合、可視範囲の長さで書式を段階的に切り替える:
   - 短い範囲（時間単位）→ 時分のみ
   - 中程度（日単位）→ 月日 + 時分
   - 長い範囲（月〜年単位）→ 月日、年月

### ツールチップの配置

**目的:** データ点の情報をカーソル近傍に表示する。画面端でもはみ出さないようにする。

**方針:** カーソル近傍に表示し、画面端では反対側に反転する。コンテナからのはみ出しをクランプして、常に見える位置に保つ。

**注意事項:**
- DOM 更新の最適化: ツールチップの内容と位置が前回と同じなら DOM の再描画をスキップする（不要なリフローの回避）

## 第9章: リアルタイムデータへの対応

### 追加データの GPU 転送

**目的:** ストリーミングデータの追加時に、最小限の転送コストで GPU バッファを更新する。

**方針:** 全データを毎回再アップロードするのではなく、追加分だけを既存バッファの末尾に書き込む。

**実装手順:**
1. 新しいデータ点を受け取る
2. GPU バッファに空きがあるか確認する
3. 空きあり（高速パス）: 追加分だけを書き込む → O(追加量)
4. 空き不足（低速パス）: 幾何成長で再確保 + 全データ再書込み → O(全量)
   ※ 低速パスの頻度は対数的に減少するため、償却コストは低い

**注意事項:**
- 変更検出のハッシュもインクリメンタルに更新する（追加分だけハッシュに反映）
- CPU 側のキャッシュ（ステージングバッファ）も同期して維持する

### 自動スクロール

**目的:** 新データ追加時に最新データを自動的に可視範囲に含める。ただしユーザーの手動操作を上書きしない。

**方針:** ズームウィンドウの右端が 100% にある場合のみ、新データ追加時に右端を 100% に追従させる。ユーザーがパン/ズームで右端を 100% 以外に移動していたら追従しない。

**理由:** ユーザーが過去のデータを参照中に勝手にスクロールされると作業が妨害される。

**代替案と棄却理由:**
- 常に最新データに追従: ユーザーの手動ズーム/パン状態が上書きされる
- 追従しない: リアルタイムモニタリング用途で新データが画面外に出てしまう
- タイマーベースの遅延追従: 実装が複雑になり、追従/非追従の判定が曖昧になる

### バウンドの動的拡張

**目的:** 追加データが既存の軸範囲を超えた場合に、軸範囲を正しく拡張する。

**方針:** 追加データが既存のバウンドを超えたらバウンドを拡張する。サンプリング済みデータではなく生データのバウンドを使う。

**理由:** サンプリングでピーク値が落ちると、軸の最大値が不正確になるため。

## 第10章: 大量データの間引き（サンプリング）

### LTTB サンプリング

**目的:** 画面のピクセル数を大幅に超えるデータ点を間引き、視覚的な形状を保ったまま描画負荷を下げる。

**方針:** LTTB（Largest-Triangle-Three-Buckets）アルゴリズムを用いる。これは Sveinn Steinarsson の論文に基づく公知のアルゴリズムである。

**実装手順:**
1. 最初と最後のポイントを必ず保持する
2. 残りのポイントを (目標点数 - 2) 個のバケットに分割する
3. 各バケットについて:
   a. 前のバケットの選択済みポイントを A とする
   b. 次のバケットの全点の平均を C とする
   c. 現バケット内の各候補 B について、三角形 ABC の面積を計算する
   d. 面積が最大の B を選択する

**理由:** 三角形面積が最大の点 = 視覚的に最も特徴的な点（ピーク、谷、急変点）。計算量 O(n) で大量データにも対応できる。

**代替案と棄却理由:**
- 等間隔サンプリング（N点ごとに1点）: ピークや谷を見逃す。視覚的な形状が大きく変わりうる
- 最大/最小サンプリング（バケット内の最大・最小を取る）: 形状は保持しやすいが、点数が2倍になるか、ピークと谷のどちらかしか取れない
- Douglas-Peucker: 計算量 O(n log n) で LTTB より遅い

**注意事項:**
- サンプリングは描画用データを減らすだけ。軸の範囲計算には必ず生データのバウンドを使う（サンプリングでピーク値が落ちると軸の最大値が不正確になるため）
- null ギャップを含むデータではサンプリングをバイパスする（ギャップ位置情報を保持する必要があるため）
- ズーム変更時は可視範囲で再サンプリングする。頻繁すぎるのでデバウンスし、可視範囲に前後マージンを持たせて小さなパンでは再サンプリングしない
- 他のサンプリング手法（平均、最大、最小）も用意すると用途に応じて選択できる
- OHLC データには専用のサンプリングが必要: 複数のローソク足を大きな時間バケットに集約し、Open = 最初の始値、High = 最大の高値、Low = 最小の安値、Close = 最後の終値として新しいローソク足を合成する

**検証基準:**
- サンプリング前後でチャートの視覚的な形状が大きく変わらないこと
- 軸の範囲がサンプリングの有無で変化しないこと
- ズームイン・ズームアウトで適切にサンプリング粒度が変化すること

## 第11章: ユーザー設定の正規化パイプライン

**目的:** ユーザーが部分的にしか指定しない設定を、描画に必要な全パラメータが確定した完全な設定に変換する。

**方針:** 以下のパイプラインで順に処理する:
1. テーマ解決: テーマ名 or オブジェクト → 具体的な色セット・フォント設定
2. デフォルトマージ: 省略フィールドにデフォルト値を補完
3. シリーズ色解決: 明示色 > シリーズ色 > パレットからの巡回割当
4. サンプリング適用: データ点数が閾値を超えたら間引き。生データも別途保持する
5. 軸バウンド計算: 生データからドメイン範囲を決定。明示指定があれば上書き
6. レイアウト調整: ズームスライダー表示時にマージンを追加

**理由:** パイプラインの各ステップが前のステップの結果に依存するため、順序を固定することで一貫した結果を保証できる。

**代替案と棄却理由:**
- ステップを任意順序で適用 → ステップ間の依存関係（例: テーマ解決前に色を決定できない）を守れず、不整合が発生する
- ユーザーに完全な設定の提供を要求する → API の使い勝手が悪くなり、ほとんどのケースで不要な設定まで指定させることになる

**実装手順:**
1. テーマは少なくともダーク/ライトの2種類を用意する。テーマオブジェクトを直接渡せるようにしてカスタマイズも可能にする
2. シリーズの色パレットは有限長。シリーズ数がパレット長を超えたらインデックスを巡回させる
3. サンプリングの閾値はユーザーが設定できるようにする。デフォルトはピクセル数を大きく超えない程度とする
4. 軸バウンドの自動計算には2つのモードを用意する:
   - 「可視範囲のデータからバウンドを導出」（Y軸のデフォルト。ズームに追従して軸範囲が変わる）
   - 「全データセットからバウンドを導出」（ズームしても軸範囲が固定）

**注意事項:**
- 非破壊: ユーザーが渡した設定オブジェクトを変更してはならない。コピーを作って作業する
- 冪等: 同じ入力に対して常に同じ出力を返す。副作用を持たない

**検証基準:**
- 空の設定オブジェクトでもエラーにならず、すべてデフォルトで描画できること
- 入力の設定オブジェクトが処理後も変更されていないこと
- 同じ設定を2回渡して同じ結果が得られること
